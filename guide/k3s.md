# k3s とは

![k3s](/guide/20_k3s.png)

k3s は、軽量化された Kubernetes ディストリビューションです。

## k3s の特徴

- **軽量**: 小規模環境でも動かしやすい
- **導入が比較的簡単**: 検証・学習を始めやすい
- **Kubernetes 互換**: 基本的な Kubernetes の概念や操作を学べる

## どんなときに使うか

- ローカル検証環境を素早く用意したいとき
- Kubernetes 学習をシンプルに始めたいとき
- エッジや小規模構成で運用したいとき

## クラスターとは

Kubernetes では、複数のマシンをまとめて 1 つのシステムとして扱います。  
このまとまりを **クラスター（Cluster）** と呼びます。

クラスターを使うことで、次のようなことが可能になります。

- コンテナを複数のマシンに分散配置できる
- 障害が起きても別のマシンで自動復旧できる
- アクセス負荷に応じてスケールできる

つまりクラスターは、

> **「コンテナをまとめて賢く管理するための基盤」**

です。

![Cluster](/guide/21_Cluster.png)

## クラスターのメリット

クラスターを使うメリットは、単に「複数台で動かせる」だけではありません。
アプリケーションを安定して運用し、継続的に改善しやすくするための土台になります。

- **耐障害性を高められる**

  - Pod や Node に障害が起きても、別の場所で再起動・再配置しやすい
  - Deployment を使うと、必要な数の Pod を維持しようとする

- **スケールしやすい**

  - アクセスが増えたときに、Pod の数を増やして負荷を分散できる
  - 使うアプリだけを個別に増やしやすい

- **移植性を高められる**

  - コンテナイメージと Manifest を使うことで、環境が変わっても同じ構成を再現しやすい
  - ローカル検証、社内環境、クラウド環境で同じ考え方を使える

- **CI/CD と相性がよい**

  - イメージのビルド、Manifest の更新、デプロイの流れを自動化しやすい
  - ローリングアップデートにより、アプリを段階的に入れ替えやすい

- **運用を標準化しやすい**

  - Deployment、Service、Ingress などの共通したリソースでアプリを管理できる
  - チーム内でデプロイや確認の手順を揃えやすい

- **リソースを効率よく使いやすい**

  - 複数のアプリを同じクラスター上で動かせる
  - 空いている Node に Pod を配置し、マシンを有効活用しやすい

![Merit](/guide/22_ClusterMerit.png)

## Container / Pod / Node の関係

Kubernetes では、コンテナをそのまま扱うのではなく、少し抽象化された単位で管理します。

- **Container**

  - 実際にアプリケーションが動く単位

- **Pod**

  - 1 つ以上のコンテナをまとめた単位
  - Kubernetes での最小実行単位

- **Node**
  - Pod を実行するマシン（物理 or 仮想）

関係としては次の通りです。

> Node の上で Pod が動き、その中で Container が動く

![Relation](/guide/23_Relation.png)

## リソースと Manifest

Kubernetes では、「何をどう動かすか」を **リソース** として定義します。  
そして、その定義を記述したファイルを **Manifest（マニフェスト）** と呼びます。

通常は YAML 形式で記述します。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-app
spec:
  replicas: 3
  ...
```

このような Manifest を適用することで、Kubernetes に対して
「こういう状態にしてほしい」と宣言します。

## 主要なリソース

最低限押さえておきたいリソースは次の 3 つです。

![Resource](/guide/24_Resource.png)

### Deployment

- Pod を管理するリソース
- レプリカ数（何個動かすか）を指定できる
- ローリングアップデートが可能

👉「アプリをどのくらい動かすか」を管理する

### Service

- Pod へのアクセス経路を提供する
- 仮想 IP（ClusterIP）を持つ
- Pod の入れ替わりを意識せずに通信できる

👉「Pod への入口」を提供する

### Ingress

- 外部からの HTTP/HTTPS アクセスを制御する
- ドメインやパスに応じて Service へ振り分ける

👉「外部公開のルーティング」を担当する

## サービス構築の流れ

Kubernetes では、これらのリソースを組み合わせてアプリケーションを公開します。

基本的な流れは次の通りです。

1. **Deployment を作成**

   - Pod（コンテナ）を起動する

2. **Service を作成**

   - Pod への内部アクセス経路を作る

3. **Ingress を作成**

   - 外部からのアクセスを Service にルーティングする

流れをまとめると、

> Ingress → Service → Pod → Container

という構造になります。

![Flow](/guide/25_Flow.png)

## k3s で学ぶメリット

k3s を使うことで、これらの仕組みを軽量な環境で体験できます。

- シンプルに構築できる
- 本番 Kubernetes とほぼ同じ概念を学べる
- 小さな環境でも十分に動作する

そのため、

> **「まず Kubernetes を触って理解する」**

という用途にとても適しています。

具体的な操作は [Hands-on の Kubernetes 編](/handson/20_kubernetes/) で確認できます。

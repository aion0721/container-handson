# コンテナとは

![OverView](/guide/10_Overview.png)

コンテナは、アプリケーションとその実行に必要な依存関係をひとまとめにした実行単位です。

## VM からコンテナへ

![VM vs Container](/guide/11_Vm.png)

従来は、アプリケーションを動かすために **仮想マシン（VM）** が使われていました。  
VM は OS ごと仮想化するため、柔軟ではあるもののリソース消費が大きいという特徴があります。

一方、コンテナはホスト OS のカーネルを共有しつつ、プロセス単位で分離された環境を提供します。

その結果、以下のようなメリットがあります。

- OS を丸ごと起動しなくていい → 軽量
- 起動が速い → 数秒〜数十秒
- 同じホスト上で多数のアプリを動かせる → 高密度

つまり、コンテナは  
**「VM の重さを解決した軽量な実行環境」** と言えます。

## 何がうれしいのか

![ContainerMerit](/guide/12_ContainerMerit.png)

コンテナには、実務でうれしいポイントがいくつもあります。

- **環境差分を減らせる**  
  開発環境と本番環境で同じイメージを使えるため、  
  「自分の環境では動いたのに…」という問題を減らせます。

- **起動が速い**  
  VM よりも軽量で、短時間で起動できます。

- **配布しやすい**  
  レジストリ経由でイメージを共有できます。

### コンテナのメリット（もう一歩深掘り）

さらに重要なポイントとして、次の 3 つがあります。

- **環境を揃えやすい（再現性）**

  - どの環境でも同じ動作を再現できる

- **起動・破棄しやすい（使い捨て可能）**

  - 必要なときだけ起動し、不要になれば削除できる

- **分離しやすい（独立性）**
  - アプリごとに依存関係を閉じ込められる
  - バージョン衝突を防げる

これらはまとめて、  
**「再現性・使い捨て・独立性」** と捉えると理解しやすいです。

## イメージとコンテナの違い

![Image](/guide/13_Image.png)

コンテナを理解する上で重要なのが「イメージ」との違いです。

- **イメージ**

  - アプリ実行に必要なファイルをまとめたもの
  - 読み取り専用（変更されない）
  - レイヤー構造で差分管理される
  - いわば「設計図」

- **コンテナ**
  - イメージから起動した実行環境
  - 書き込み可能なレイヤーを持つ
  - 実際にプロセスとして動作する
  - いわば「実体」

つまり、

> イメージ = 静的なもの  
> コンテナ = 動的に動くもの

という関係になります。

## 基本的なライフサイクル

コンテナの基本的な流れは次の通りです。

1. イメージを取得またはビルドする
2. イメージからコンテナを起動する
3. 動作確認する
4. 停止・削除する

![Lifecycle](/guide/14_Lifecycle.png)

## コンテナの稼働の流れ

![Flow](/guide/15_Flow.png)

実務では、次のような流れでコンテナを扱います。

1. レジストリ（例: Docker Hub）からイメージを取得（Pull）
2. イメージを基にコンテナを起動（Run）
3. アプリケーションがプロセスとして動作する
4. 不要になったら停止・削除する

また、必要に応じて自分でイメージを作成します。

- Dockerfile または Containerfile を記述する
- イメージをビルドする（Build）
- レジストリにアップロードする（Push）

まとめると、次の 2 パターンになります。

- **Pull → Run**
- **Build → Push → Pull → Run**

## コンテナだけでは足りない？

ここまでで、コンテナ単体の便利さは理解できたと思います。

しかし実際のシステムでは、次のような課題が出てきます。

- コンテナの数が増えると管理が大変
- どのサーバで動かすかを考える必要がある
- 障害時の再起動やスケーリングを自動化したい
- ロードバランシングをしたい

こうした課題を解決するために登場したのが  
**コンテナオーケストレーション** です。

## 次の話題: オーケストレーション

コンテナを複数まとめて運用するには、起動する場所、数、再起動、通信経路などを管理する仕組みが必要になります。
この仕組みを **コンテナオーケストレーション** と呼びます。

代表的なコンテナオーケストレーションツールが **Kubernetes（K8s）** です。

このハンズオンでは、Kubernetes を軽量に使える `k3s` で、コンテナをどのように管理するかを体験します。
